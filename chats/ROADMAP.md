一个 Dockerfile 构建，运行即开始。

整体上代码部分用 Node + Python 双架构，Dockerfile 基于 Alpine 构建。

自动爬取若干免费 proxy 的网站，爬取脚本用 plugins 自动管理

所有代理储存在数据库中，储存的信息有：

1. 协议（http https socks4 socks5）

1. IP

1. 端口

1. 短名称（一般是国家代号+城市名称）

1. 长名称（一般是国家全称+城市名称）

1. 备注

---

2. hash，是 协议://ip:端口 的sha256的后7位，用于唯一标识节点。

2. 第一次添加的时间

2. 最后一次添加的时间

2. 直接 ping 其 ip 的延迟

2. 直接请求 https://www.google.com/generate_204 的延迟

2. 直接请求 http://www.msftconnecttest.com/connecttest.txt 的延迟

2. 请求 https://speed.cloudflare.com/__down?bytes=10000000 的下载速度

支持添加 plugins，里面支持 Javscript Typescript Python 三种语言的脚本，每个脚本你设计一个接口，每次调用他们，返回一个列表表示所有代理，包含上面所有编号为 1 的内容（如果没有按照空处理），注意返回的名称不包含 hash 需要你自己加上。

注意：插件是不能外部添加的也就是说，本地的 plugins 文件夹就是编译时放到目标中的所有插件，不能动态添加，要添加只能重新构建。

在 config.yml 中指定 plugin 以及定时策略（cron 字符串），按照计划进行执行，在文档中建议配置多个 plugin 错开时间。

到时间后，通过 plugin 获取节点列表，然后自动获取他们的hash，添加到一个“准备列表”中。

准备列表是一个用于准备进行测试的列表，其中的代理是仅仅通过 hash 去重的，没有经过实际验证。

然后以一个较低的频次对准备列表进行请求，按照每个任务完成再等待 1s，单线程，再进行下一个任务，以在配置较差的环境下运行。

1. 验证是先 ping，如果失效（失败或者延迟 >5000ms），那么直接从准备列表中删除。

1. 如果 ping 通过，进行请求两个真延迟，用这个代理进行请求，获取延迟，如果两个都失败或所有延迟>5000ms那么也从准备列表中删掉。

以上所有操作，最大3次重试，注意如果超过 5000ms 没有相应，那么强制停止。

对于删掉的内容，维护一个“删除列表”，只记录hash和删除时间，下一次如果接收到，自动看hash如果有这个在24h内测试不通过，那么不再测试，直接跳过。

通过上面测试的内容，放到一个“可用列表”中，表示代理可用。

然后进行一个测速，测速的速率再低，每次从可用列表里面找一个没有测试过的、延迟最低的进行，请求 cloudflare 的文件，获取下载速度，每次请求最大 10MB 且 10s 以内，如果 10s 没有请求完那么中断，按照下载的大小/下载的用时算速度，记录。对于所有速度>=10MB/s的放到“高性能列表”中。

以上是处理新增的节点，还要处理已经有的节点的如果它失效了删除的操作。

- 对于每一个添加到可用列表超过 8h 的节点，每 8h 验证一下，验证方法是先请求 ping 再请求真延迟，超过 7500ms 就从可用列表中删除，注意这个不加到删除列表中。

- 对于每一个加到高性能列表的节点，每 8h 验证一下，做速度测试，如果 <1MB/s 那么剔除。

以上是节点的维护操作。以上基本假设是，传进来的节点很多，可能有成千上万，但是可以用的相对较少，只有大约 10%～20%。

然后做一个前端和外部 api 接口，暴露到 8080 端口（可以通过 ENV 改），维护一个 /api... 以及子路由等等做请求，可以请求前面提到的各种在应用环境中有用的东西，返回是 JSON。支持订阅转换，用 acl4ssr 的那个 subconverter 作为后端，支持 ENV 设置自定义后端地址。

再做一个前端，监控前面说到的各种东西，一个 dashboard 和若干列表式的查看，注意优化性能，所有显示或者传送到客户端的，限制每屏、每请求最大返回数量（有可能极限情况上万个可用节点）。

目前先实现上面的总体框架和两个plugin即可，这两个plugin用python写，做一定防反爬处理
注意：我的主机平台不支持 docker compose，因此只能构建一个 Dockerfile 单挂载使用，所有逻辑必须放在里面，同时不要调用外部数据库，所有数据库基于本地的目录，便于我挂载长期使用。

同时左上关闭docker后自动读取配置恢复的逻辑，便于我更新和维护。我远程的内存可调，但是默认是 64MB，如果太少了，我可以调到128MB 256MB，但是我希望默认用的少一点，避免花费太大。
